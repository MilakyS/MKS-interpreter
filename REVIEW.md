# 🦍 MKS Senior Review & Optimization Report

Этот отчет содержит результаты глубокого анализа интерпретатора MKS и описание внесенных улучшений.

## 🚀 Что было сделано (Оптимизации)

### 1. Ускорение Environment (Хеш-таблиц)
* **Проблема**: Для вычисления индекса бакета использовался оператор `%` (остаток от деления). Это дорогая операция для процессора.
* **Решение**: Поскольку размер хеш-таблицы в MKS всегда является степенью двойки (8, 16, 32...), мы заменили `% bucket_count` на `& (bucket_count - 1)`.
* **Результат**: Быстрый побитовый доступ к переменным.

### 2. Оптимизация Лексера
* **Проблема**: В функции `Read_Number` на каждое встреченное число вызывался `malloc`, `memcpy` и `free`. В циклах это создавало колоссальную нагрузку на кучу.
* **Решение**: Теперь `strtod` читает число напрямую из исходного буфера. Лишние аллокации полностью устранены.
* **Результат**: Меньше мусора для GC и быстрее парсинг.

### 3. Исправление критического бага в GC
* **Проблема**: При создании массивов (`[1, 2, 3]`), элементы вычислялись рекурсивно, но поле `count` устанавливалось только в самом конце. Если GC запускался во время вычисления второго или третьего элемента, первые элементы могли быть удалены как "недостижимые".
* **Решение**: Теперь поле `count` обновляется инкрементально (`count = i + 1`) сразу после вычисления каждого элемента.
* **Результат**: Гарантированная стабильность при работе с большими массивами.

## 🛡️ Безопасность и Стабильность

### 1. Защита от переполнения стека в GC
* Алгоритм маркировки окружений (`gc_mark_env`) теперь итеративный, а не рекурсивный. Это предотвращает `Stack Overflow` при очень глубокой вложенности функций или длинных цепочках замыканий.

### 2. Информативные ошибки
* Ошибки деления на ноль теперь выводят номер строки, где произошел сбой, что значительно упрощает отладку MKS-скриптов.

## 🧠 Советы на будущее (Full Senior Feedback)

1. **Интернирование строк**: Сейчас при каждом сравнении строк вызывается `strcmp`. Если добавить таблицу интернирования (String Pooling), сравнение строк станет `O(1)` (сравнение указателей).
2. **Байт-код**: Текущий древовидный интерпретатор (Tree-walking) медленный по определению. Переход на Bytecode VM даст прирост в 5-10 раз.
3. **NaN Boxing**: Для `RuntimeValue` можно использовать технику NaN Boxing (хранение всех типов внутри одного `double`). Это уменьшит размер структуры с 24-32 байт до 8 байт, что критически важно для кеша процессора.
4. **Error Handling**: Сейчас используется `exit(1)`. Для серьезного языка стоит перейти на механизм `longjmp` или возврат кодов ошибок, чтобы интерпретатор мог восстанавливаться после сбоев.
5. **Small Object Optimization**: В `Environment` можно добавить массив из 4-8 бакетов прямо в структуру (inline), чтобы не вызывать `malloc` при создании короткоживущих локальных областей видимости.

## ⚠️ Глубокий разбор Stack Overflow в GC

### Что вызывает переполнение стека?
В GC переполнение стека (Stack Overflow) происходит из-за **рекурсивной маркировки** глубоких структур данных.

#### 1. Цепочки окружений (Environment Chain)
Если в MKS создать глубокую вложенность функций или цепочку замыканий, поле `parent` в `Environment` сформирует длинный список.
**MKS код:**
```rust
fnc deep(n) ->
    if (n ?= 0) -> return fnc() -> return 0; <-; <-
    var inner =: deep(n - 1);
    return fnc() -> return inner(); <-;
<-
var boom =: deep(5000); // Создает 5000 вложенных окружений
```

**C код (до фикса):**
```c
static void gc_mark_env(Environment *env) {
    if (env == NULL || ((GCObject*)env)->marked) return;
    ((GCObject*)env)->marked = true;
    // РЕКУРСИВНЫЙ ВЫЗОВ — корень зла
    gc_mark_env(env->parent);
}
```
*Каждый вызов `gc_mark_env` съедает фрейм на системном стеке C. При 5000+ вложенностях стек ОС просто заканчивается.*

#### 2. Длинные списки на массивах
Аналогично для массивов, если использовать их как связанные списки.
**MKS код:**
```rust
var list =: [0];
for (var i =: 0; i < 10000; i =: i + 1) ->
    list =: [i, list];
<-
```
**C код (текущий риск в `gc_mark_object`):**
```c
case GC_OBJ_ARRAY: {
    const ManagedArray *arr = (ManagedArray *)obj;
    for (int i = 0; i < arr->count; i++) {
        gc_mark_value(&arr->elements[i]); // Рекурсия здесь!
    }
    break;
}
```

### Решение: Итеративная маркировка (Mark-Stack)
Вместо рекурсии используйте явный стек (массив) или очередь для объектов, которые нужно пометить. Для `Environment` я уже внедрил `while (env != NULL)`, что решило проблему для цепочек окружений.

## 🛠️ Стоит ли добавлять `malloc` и `free`?

**Короткий ответ: Для "языка для души" — скорее нет, чем да.**

**Почему "Нет":**
1. **Безопасность**: Главная фишка языков с GC — отсутствие `use-after-free` и `double free`. Добавляя ручное управление, ты лишаешь пользователя этой гарантии.
2. **Сложность**: Тебе придется объяснять пользователю, почему одни вещи удаляются сами (строки, массивы), а другие — нет.
3. **Конфликты**: Если пользователь сделает `free` объекта, на который еще смотрит GC, всё упадет.

**Почему "Да" (как эксперимент):**
- Если ты хочешь сделать MKS максимально "низкоуровневым" и дать пользователю полный контроль над памятью (например, для системного программирования).
- Как обучающий опыт: реализация `malloc` поверх системного аллокатора — отличная задача.

**Senior-совет:** Вместо `malloc/free` лучше реализуй **Native Modules** на C, где ты сможешь выделять память вручную внутри C-функций, а в MKS возвращать "Handle" (указатель), обернутый в GC-объект. Это сохранит чистоту языка и даст мощь C там, где она нужна.
